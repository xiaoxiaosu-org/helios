# ADR-20260228: CAP-007 技术债追踪机制落地

## 状态
Accepted

## 日期
2026-02-28

## 背景
- 执行计划中 CAP-007 要求建立统一技术债清单与偿还机制。
- 现有清单仅满足“存在 TD- 记录”，缺少 owner、更新时间与状态治理约束，难以支撑持续跟踪。

## 决策
- 扩展 `docs/02-架构/技术债清单.md`，引入 `Owner`、`最近更新`、`验收标准` 字段。
- 增加“在制技术债 / 已完成”双分区，并补维护机制（评审更新、完成迁移、超期说明）。
- 强化 `scripts/cap/CAP-007/verify.sh`，将关键字段与状态合法性纳入 CAP 验收。
- 将路线图 `CAP-007` 状态更新为 `Done`。

## 影响

### 正面影响
- 技术债清单从“静态列表”升级为“可治理队列”，可分配 owner 并追踪更新节奏。
- CAP 验收可在最小范围内阻断“有表无治理规则”的退化。

### 负面影响
- 清单维护成本上升，计划评审时需要同步维护状态与更新时间。
- 目前仍是 CAP 级验收，尚未接入常规 CI，全局强制性需后续迭代。

## 备选方案
- 继续保留最小清单格式：拒绝，无法满足 owner/验收/状态治理要求。
- 直接引入外部 issue tracker：暂不采用，先以仓库内文档与脚本实现低成本闭环。

## 版本影响
- 是否影响 API：否
- 是否影响事件模型：否
- 是否影响数据结构：否
