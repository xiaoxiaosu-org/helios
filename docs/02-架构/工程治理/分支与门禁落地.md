# 分支与门禁落地（让 CI 真正阻止“不合格合并”）

本仓库的门禁（Workflows、脚本、模板）要“真正起作用”，关键在于 **保护主干分支**：允许推送 feature 分支，但禁止绕过 PR 直接把不合格变更进入主干。

结论：
- `git push` 不会被 CI 直接拦住（它只会触发 CI 产出状态）。
- **分支保护（Branch protection）** 才是“阻止合并”的开关：未通过门禁的变更无法进入受保护分支（通常是 `main`）。
- 为减少“推上去才发现失败/提交信息不规范”，建议启用本地 hooks（`pre-commit` + `commit-msg` + `pre-push`）与提交模板。

---

## 1) GitHub 必配：保护主干（必须）

推荐使用 **Rulesets**（你截图里的界面），位置通常在：Settings → Rules → Rulesets。
（若仓库仍在用旧入口，也可在 Settings → Branches → Branch protection rules 做同等配置。）

### 1.1 保护 `main`（或你的默认主干）

Target branches：
- Branch targeting criteria：`main`

Rules（按你界面逐项对应，推荐勾选）：
- Require a pull request before merging
  - Required approvals：`1`
  - Dismiss stale pull request approvals when new commits are pushed
  - Require conversation resolution before merging
  - （可选）Require review from specific teams：当你们已经有稳定的 Owner 团队与路径分配时再启用
  - （可选）Require approval of the most recent reviewable push：当你们希望“最后一次 push 也必须被别人 approve”再启用
- Require status checks to pass
  - Status checks that are required：勾选 `quality-gates`、`checks`
- Require branches to be up to date before merging
- Block force pushes
- Allowed merge methods：建议只允许 `Squash`（或只允许 `Rebase`，二选一），不要允许 merge commit
- Require linear history（配合上面的 Squash/Rebase）

不建议现在就勾选（容易把流程卡死/需要额外体系支撑）：
- Restrict creations / Restrict updates / Restrict deletions（除非你已经配置好 bypass 权限与运维流程）
- Require deployments to succeed（除非你已有稳定的部署环境与对应 workflow）
- Require signed commits（除非团队已统一签名与密钥管理）
- Require code scanning results / Require code quality results（除非已启用对应扫描并稳定产出结果）

这样配置后：
- 你依然可以随意 `git push` 到 feature 分支；
- 但 **未通过门禁的变更无法合并进主干**。

## 2) 推荐分支管理流程（日常）

### 2.1 分支命名

建议约定：
- CAP 推进分支：`cap/CAP-00X-<topic>`
- 常规功能：`feat/<topic>`
- 修复：`fix/<topic>`
- 紧急修复：`hotfix/<topic>`（见治理细则的 Hotfix 例外流程）

### 2.2 日常流程（feature → PR → merge）

1) 从主干拉新分支
- `git checkout main`
- `git pull --rebase`
- `git checkout -b cap/CAP-009-pr-template`

2) 按“每个 PR 只推进一个 CAP”的原则实现最小变更

3) 本地跑一次门禁（推荐）
- `scripts/cap/verify.sh CAP-00X`
- 如涉及业务代码目录（`modules/`、`apps/`、`services/`、`src/`）：`scripts/ci/verify.sh`

4) 推送分支并创建 PR（base = `main`）
  - 推送前先给出“本次改动摘要”（提交列表 + 文件列表），确认后再执行远端 push

5) 等待 CI 通过 + Owner Review 通过

6) 合并策略（建议二选一）
- Squash merge：更干净的主干历史（推荐）
- Rebase merge：要求线性历史时使用

7) 合并后删除分支

## 3) 推荐增强：本地 hooks + 提交模板（减少返工）

本地 hooks 的定位：**提前失败**，节省 CI 循环时间；它不是安全边界，安全边界仍由受保护分支 + 必过 checks 保证。

启用方式：
- 执行：`scripts/dev/install-git-hooks.sh`
- 启用后会自动配置：
  - `core.hooksPath=.githooks`
  - `commit.template=.github/commit_message_template.txt`

启用后行为：
- `pre-commit`：禁止在 `main` 上直接提交（鼓励一律分支 + PR）
- `commit-msg`：每次提交必须满足
  - 标题格式：`type(scope): summary`（支持 `!` 破坏性标记）
  - 标题类型：`feat|fix|docs|style|refactor|perf|test|build|ci|chore|revert`
  - 正文必须包含段落：`Why:`、`How:`、`Verify:`
- `pre-push`：在 `git push` 前自动运行
  - 默认禁止直接更新 `main`（分支为 `main` 或 push 目标为 `refs/heads/main`）
  - 自动输出待推送摘要（提交列表 + 文件列表）
- `scripts/docs/index-check.sh`
- `scripts/ci/verify.sh`
- 若当前分支名包含 `CAP-00X`，则额外运行：`scripts/cap/verify.sh CAP-00X`

临时跳过（仅紧急场景；需在 PR 说明原因）：
- `HELIOS_ALLOW_RELAXED_COMMIT_MSG=1 git commit ...`（仅放宽本次提交信息校验）
- `HELIOS_ALLOW_PUSH_MAIN=1 git push ...`（仅放宽本次 main 直推拦截，仍可能被远端 ruleset 拒绝）
- `git push --no-verify`

## 4) CI 强制项（防止绕过本地规则）

- `doc-check`（`checks`）会校验新增提交标题格式：`type(scope): summary`
- `doc-check`（`checks`）会校验 PR 描述是否按模板填写：
  - 必须包含模板全部段落
  - `变更摘要（Summary）` 不能为空
  - `变更类型（Type）` 至少勾选一项
- `doc-check`（`checks`）会校验规则文件：
  - `AGENTS.md` 与 `docs/02-架构/工程治理/**` 仅允许 `.md`
  - 规则文件内容需包含中文描述
- 即使本地绕过 hook，未满足上述规则仍无法通过合并门禁
